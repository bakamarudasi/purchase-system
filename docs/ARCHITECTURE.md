# アーキテクチャ概要

このプロジェクトは Google Apps Script (GAS) で動作する購入申請システムを TypeScript で管理しています。主要な責務の分離は以下の通りです。

## エントリーポイント (`src/main.ts`)

- GAS から公開されるグローバル関数を定義します。
- 画面や外部システムから呼ばれる Web API の入口として機能し、**ビジネスロジックは直接書かずにサービス層へ委譲**します。
- 例: `addApplication` や `approveApplication` などは `ApplicationService` を呼び出しており、ここで行うのは入力値の受け渡しとログ出力・例外処理のみです。

> 申請フローを新しく公開したい場合は、`main.ts` に新しいエクスポート関数を追加し、その中でサービス層のメソッドを呼び出してください。

## サービス層 (`src/services/`)

- スプレッドシートや Drive など GAS API との連携、およびドメインロジックを担います。
- 既存の申請処理は `ApplicationService.ts` にまとまっており、データの取得・作成・更新・承認ステータス変更などをここで実装します。
- ファイルの保存は `FileService.ts`、ユーザー関連は `UserService.ts` に切り出されています（`index.ts` はサービスをまとめてエクスポートするためのハブです）。

> 申請データの作成やフローの分岐を実装する場合は、`ApplicationService` にメソッドを追加するか、責務が増える場合は新しいサービスファイルを作成し、`src/services/index.ts` からエクスポートしてください。

## モデル (`src/models/`)

- 申請データ (`Application`) など、ドメインオブジェクトの型定義を提供します。
- サービス層で扱うデータ構造を共通化し、`main.ts` からも型安全に利用できます。

## 設定 (`src/config.ts`)

- シート名や列インデックス、ステータス定義などを集中管理しています。
- 環境固有の値（フォルダ ID など）をここで管理することで、サービス層のコードが読みやすくなります。

## ユーティリティ (`src/utils/`)

- 日付フォーマットや型変換など、汎用的な処理を配置します。
- 申請フローの実装中に共通化できる処理が出た場合はユーティリティとして切り出すと良いでしょう。

## 実装の流れ

1. **新しいフローを公開したい**: `main.ts` に GAS から呼び出される関数を追加する。
2. **ビジネスロジックやデータ操作を実装する**: `ApplicationService.ts` などサービス層にメソッドを実装する。
3. **データ型が増える場合**: `src/models/` に型を追加し、サービス・エントリーポイントから利用する。
4. **設定値が必要な場合**: `src/config.ts` に定数を追加する。

この分離に従うことで、GAS エントリーポイントは薄く保たれ、テストや保守がしやすい構造になります。
