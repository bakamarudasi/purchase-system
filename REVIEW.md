# コードレビュー所見

## ✅ 良い点
- GASのグローバル公開関数を `global` 経由で束ねており、`main.ts` がハブとして整理されています。
- `ApplicationService` がスプレッドシートとのデータ授受とビジネスロジックを担う構造は責務が明確です。
- 行データ→モデル変換の際に `safeParseInt` / `safeParseFloat` を使い、型崩れへの耐性を持たせています。

## ⚠️ 気になる点
1. **添付ファイルのバリデーション未実装**  
   `config.ts` で `ALLOWED_MIME_TYPES` や `MAX_FILE_SIZE` を定義していますが、`ApplicationService.addApplication` ではそれらを参照しておらず、どの MIME/サイズでも Drive に保存されてしまいます。大きなファイルや想定外のファイル形式を弾けるよう、保存前に `Utilities.base64Decode` したバイト長の確認や MIME の照合を行うと安全です。

2. **コメントの残骸**  
   `ApplicationService.ts` 冒頭のコメントアウト行に "将来用に宣言ぐらいはしておく'" というメモが残っています。レビュー観点としては、不要コメントは削除する／`TODO` に書き換えるなどして、他メンバーにも意図が伝わる形にした方が良いです。

3. **`getApproverList` の列指定とコメントの齟齬**  
   コメントでは "A列の2行目から" となっていますが、実際には `getRange('B2:B')` で B 列を読みに行っています。シートの実データと齟齬があると承認者リストが常に空になる可能性があるので、列を統一するかコメントを合わせるかを検討してください。

4. **ファイル情報取得時の例外**  
   `parseFileInfo` で Drive からファイルを再取得していますが、承認者が閲覧権限を持たない場合は `DriveApp.getFileById` が例外を投げます。現在は catch して最低限の情報を返していますが、閲覧者の権限によっては常に例外経路に入りパフォーマンスが低下します。権限設計（共有フォルダや `DriveApp` での権限委譲）を確認するか、URL 文字列から必要最低限の情報だけ組み立てるなどの対策を検討してください。

5. **`getCurrentUser` の名称解決**  
   GAS の `Session.getActiveUser()` は社内ドメインでは空文字になるケースがあります。名前解決に失敗した際のフォールバック（例: Apps Script プロパティサービス経由での辞書、もしくはメールアドレスそのものを返すなど）を検討すると UX が向上します。

## 💡 改善案
- 添付ファイル処理に関する単体テスト（`test.ts` など）を追加し、サイズ／MIME バリデーションの挙動を検証できるようにする。
- ステータス値に `購入済` や `完了` が含まれていますが、更新処理（`updateApplicationStatus`）では `STATUS.PENDING/APPROVED/REJECTED` しか使っていません。将来的に購入完了の遷移を導入する予定があるなら、サービス層の処理も合わせて準備しておくとよいでしょう。
